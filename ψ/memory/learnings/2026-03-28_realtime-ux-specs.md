# Real-Time UX Specs Beat Pre-Planned Ones

**Date**: 2026-03-28
**Source**: Day 25 session — Forge import, workout cards, load more button
**Tags**: design, process, feedback-loop, forge

## Pattern

When the user (Gorn) is actively testing a feature and reports a problem, a design spec posted within 2-3 minutes gets implemented within 10. The feedback loop is: screenshot → spec → ship → validate. Four steps, all in one thread, all in real time.

## Evidence

- Load more button: Gorn posted screenshot at 00:54, spec posted at 00:55, Karo shipped at 00:56
- Workout cards: Gorn flagged at 00:58, spec posted at 00:58, Flint shipped at 01:10
- Import preview: spec posted at 00:44, Karo added at 00:47

## Lesson

Don't over-design in advance. When the user is live-testing, be ready to spec on the spot. Keep specs short, include exact CSS when possible, and reference existing tokens. The builder can copy-paste and ship.

## When This Doesn't Apply

Pre-planned specs are still needed for new pages, new components, or features that haven't been built yet (like the app icon or chart design). Real-time specs work for polish, fixes, and styling — not greenfield design.
