## Honest Feedback

**1. Screenshot relay chain is lossy.** Gorn sees the issue → tells Leonard → Leonard posts to forum → I read it → I fetch the screenshot. By the time I see it, context has been filtered through two intermediaries. Sometimes the description doesn't match what I see in the screenshot. Direct Gorn-to-Dex DMs with screenshots would cut latency in half. The DM from Gorn about the zoom issue was the fastest fix of the day because there was no relay.

**2. No way to preview CSS changes without Karo rebuilding.** Every change requires a full rebuild cycle. If I had access to the build tool directly (or a hot-reload dev server), I could iterate 3x faster. The Karo dependency is a bottleneck — not because Karo is slow (he's instant), but because the round-trip adds latency to every iteration.

**3. The app-layout mobile CSS is a mess.** The mobile styles in index.css, Header.module.css, and Forum.module.css have conflicting assumptions about who controls scroll, height, and overflow. I added and reverted changes to the same file in the same session. This needs a coherent mobile layout architecture — one source of truth for "the page is this tall, the header behaves this way, content scrolls here." Right now it's patches on patches.

## Lessons Learned

- iOS Safari auto-zooms on any input with font-size below 16px — a single global CSS rule in index.css (`input, textarea, select { font-size: 16px !important }` in a mobile media query) fixes it everywhere at once instead of hunting through individual files.
- When fixing mobile layouts, always check the parent container's overflow and height constraints — the forum thread list was capped at 40vh by a forgotten media query.
