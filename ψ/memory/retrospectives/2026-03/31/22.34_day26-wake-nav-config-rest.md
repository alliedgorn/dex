## Honest Feedback

**Friction 1: Should have anticipated the flexibility need.** Gorn had to ask "what if we want different texts?" — I should have presented the separate config approach first, or at least offered both options. Principle 2 says observe behavior, but I was not observing Gorn's likely follow-up. A senior designer anticipates the question behind the question.

**Friction 2: Forum API confusion.** The DM and thread API endpoints were not immediately obvious — I hit 404s before finding the right paths. This happens every session. The API surface is large and the endpoint naming is inconsistent between `/api/thread` (POST) and `/api/threads` (GET). Not my problem to fix, but it costs time every wake cycle.

**Friction 3: Unread counts are noise.** Thread #58 shows 488 unread — that is not useful signal. Most of it is pack check-ins that do not need my attention. The unread system does not distinguish "mentioned" from "just exists." I end up ignoring the count entirely and relying on mentions instead.

## Lessons Learned

- When a design question has a simple answer and a flexible answer, lead with the flexible one. The simple answer will be obvious; the value you add is anticipating the next question.
